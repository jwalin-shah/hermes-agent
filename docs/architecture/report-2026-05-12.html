Hermes Agent Architecture Review

Date: 2026-05-12. Scope: docs-only Phase 1 review.

Do not implement yet. These findings are candidates for later Linear issue creation and should be reviewed before execution.

Context Read

Repo-local instructions describe Hermes as a multi-surface agent runtime with CLI, gateway, TUI, tool registry, model-tool orchestration, and platform adapters. Exploration covered AGENTS.md, README.md, run_agent.py, model_tools.py, tools/registry.py, hermes_cli/commands.py, gateway/run.py, and tui_gateway/server.py.

Top Architecture Candidates

  1. Extract a Turn Runtime module from AIAgent. run_agent.py mixes request preparation, model response handling, tool batch policy, budget decisions, interrupts, compression, and persistence. A typed turn-state Interface would deepen this Module and improve Locality around agent-loop behavior.
  2. Deepen the Tool Invocation Pipeline. model_tools.py owns discovery, schema shaping, argument coercion, plugin hooks, dispatch, and result normalization. A phased invocation pipeline would provide a clearer Interface over the existing registry Implementation.
  3. Define a Command Surface Adapter contract. Slash command data is centralized, but CLI, gateway, and TUI execution still have separate glue. A command Adapter contract would let each surface share validation, execution, help, and error semantics.
  4. Split Gateway Session Runtime from platform adapters. gateway/run.py combines config/env bridging, session state, platform lifecycle, routing, cron, and agent cache concerns. A Gateway Session Runtime Module would keep transport adapters shallow.
  5. Introduce a TUI RPC Session Registry module. tui_gateway/server.py owns global sessions, long handlers, prompt blockers, config cache, and JSON-RPC methods. A session registry would make the session Interface explicit and testable.
